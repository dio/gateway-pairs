# Design

## Summary

`gateway-pairs` deploys N isolated Envoy Gateway controller+dataplane pairs
inside a single Kubernetes cluster. Each pair is one Helm release of `eg-pair`.
CRDs are installed once cluster-wide via `eg-crds` + `hack/install-crds.sh`.

---

## Two-namespace model per pair

```
tr-system-{i}      Helm release Secret
                   EG controller Deployment
                   EnvoyProxy CR  ← GatewayClass.parametersRef points here
                   HTTPRoutes     ← tenants write here

tr-dataplane-{i}   Gateway        ← EG puts proxy in Gateway's namespace
                   Envoy proxy Deployment + Service  (generated by EG)
```

**Why Gateway lives in the dataplane namespace:** In GatewayNamespace mode EG
places the proxy Deployment in the Gateway object's namespace. Putting the
Gateway in `tr-dataplane-{i}` isolates the proxy there, away from the
controller in `tr-system-{i}`.

**Why HTTPRoutes live in the system namespace:** That is where the fraser/transit
control plane writes them. The Gateway's `allowedRoutes` uses `from: Selector`
targeting `tr-system-{i}` to permit cross-namespace attachment.

**Why EnvoyProxy lives in the system namespace:** It is control-plane config --
owned and written by the operator alongside the controller. The GatewayClass
`parametersRef` points into `tr-system-{i}`.

---

## Install

```bash
helm upgrade --install eg-pair-1 ./charts/eg-pair \
  --namespace tr-system-1 --create-namespace \
  --set pair.index=1 --skip-crds
```

`tr-system-{i}` is both the Helm release namespace and the controller namespace.
`tr-dataplane-{i}` is a chart-declared Namespace resource, Helm-owned, removed
on uninstall.

## Namespace naming

All names derive from three chart values: `pair.namePrefix`, `pair.index`, and
`pair.nameSuffix`.

**Rules:**
- `namePrefix`: auto-appends `-` if non-empty and not already ending with one.
- `nameSuffix`: appended as-is -- the caller controls the separator (including none).
  Useful for environment (`prod`, `staging`) or region (`eu-west`) identifiers.
- `nameSuffix` empty + `index > 0` → suffix defaults to `-{index}`.
- `nameSuffix` empty + `index == 0` → no suffix (single/unnamed pair).
- GatewayClass name omits the role fragment (`system`, `dataplane`) -- just prefix+suffix.

**Examples:**

| namePrefix | index | nameSuffix | systemNS | dataplaneNS | GatewayClass |
|---|---|---|---|---|---|
| `tr` | `1` | `""` | `tr-system-1` | `tr-dataplane-1` | `tr-1` |
| `myapp` | `1` | `""` | `myapp-system-1` | `myapp-dataplane-1` | `myapp-1` |
| `myapp` | `0` | `""` | `myapp-system` | `myapp-dataplane` | `myapp` |
| `myapp` | `0` | `-prod` | `myapp-system-prod` | `myapp-dataplane-prod` | `myapp-prod` |
| `myapp` | `0` | `-eu-west` | `myapp-system-eu-west` | `myapp-dataplane-eu-west` | `myapp-eu-west` |
| `myapp` | `1` | `1` | `myapp-system1` | `myapp-dataplane1` | `myapp-1` |
| `""` | `1` | `""` | `system-1` | `dataplane-1` | `1` |

The `nameSuffix` field gives full control when the default `index`-based naming
doesn't fit. Common patterns:

```bash
# Numeric pair (default)
--set pair.index=1

# Environment-named pair
--set pair.namePrefix=myapp --set pair.index=0 --set pair.nameSuffix=-prod

# Region-named pair
--set pair.namePrefix=myapp --set pair.index=0 --set pair.nameSuffix=-eu-west

# Custom prefix
--set pair.namePrefix=myapp --set pair.index=1
```

## allowedRoutes wiring

The Gateway in `tr-dataplane-{i}` permits HTTPRoutes from `tr-system-{i}`:

```yaml
listeners:
- name: http
  port: 80
  protocol: HTTP
  allowedRoutes:
    namespaces:
      from: Selector
      selector:
        matchLabels:
          kubernetes.io/metadata.name: tr-system-1
```

`kubernetes.io/metadata.name` is automatically set by Kubernetes on every
namespace. No extra labeling needed.

HTTPRoutes in `tr-system-{i}` reference the Gateway with an explicit namespace:

```yaml
parentRefs:
- name: eg
  namespace: tr-dataplane-1
```

## Controller isolation: unique controllerName per pair

Each pair's EG controller must have a unique `controllerName`. If N controllers
share the upstream default (`gateway.envoyproxy.io/gatewayclass-controller`),
each controller tries to reconcile all N GatewayClasses and fails with:

```
failed to find envoyproxy: unknown namespace for the cache
```

The chart derives the controllerName from the GatewayClass name so it is always
unique per pair:

```yaml
# ConfigMap envoy-gateway-config:
gateway:
  controllerName: gateway.envoyproxy.io/tr-1

# GatewayClass:
spec:
  controllerName: gateway.envoyproxy.io/tr-1
```

Both are generated from the same `_helpers.tpl` value -- they cannot drift.

---

## Watch list

Both namespaces must be in the watch list:

```yaml
watch:
  type: Namespaces
  namespaces:
  - tr-system-1     # controller reads its own TLS secret + manages EnvoyProxy, HTTPRoutes
  - tr-dataplane-1  # controller manages Gateway + generated proxy resources
```

Omitting `tr-system-{i}` causes Gateways to be `Accepted=True` but never
`Programmed=True` (controller cannot read its own TLS cert).

---

## Authentication: mTLS → JWT

Gateway Namespace mode shifts xDS auth from mTLS to JWT:

- Proxy pods (in `tr-dataplane-{i}`) authenticate using projected ServiceAccount
  JWT tokens (short-lived, auto-mounted).
- Controller validates via `TokenReview` API. Requires `tokenreviews/create` at
  cluster scope.
- Only the CA certificate is available in `tr-dataplane-{i}`. No client certs.

If missing: `tokenreviews.authentication.k8s.io is forbidden` in controller logs.

---

## RBAC: infra-manager in both namespaces

The controller creates resources in both namespaces:
- `tr-system-{i}`: proxy ServiceAccount (`eg`)
- `tr-dataplane-{i}`: proxy Deployment, Service, ConfigMap

The `infra-manager` Role + RoleBinding must exist in both.

---

## Uninstall sequence

The proxy pod has a finalizer that only the running controller can clear.
Delete the Gateway first:

```bash
kubectl delete gateway eg -n tr-dataplane-1 --wait=true --timeout=60s
kubectl wait --for=delete deployment \
  -l gateway.envoyproxy.io/owning-gateway-name=eg \
  -n tr-dataplane-1 --timeout=60s
helm uninstall eg-pair-1 -n tr-system-1
# tr-system-1 and tr-dataplane-1 terminate cleanly
```

Skipping the Gateway deletion causes `tr-dataplane-{i}` to hang in `Terminating`.

---

## Merged Gateways incompatibility

Gateway Namespace Mode is **not supported** with Merged Gateways.

---

## Planned: subchart migration

Add `gateway-helm` as a Helm subchart dependency. `eg-pair` becomes a thin
wrapper: passes `deploy.type: GatewayNamespace`, `controllerName`, and watch
list as values; adds GatewayClass, EnvoyProxy, Gateway, and namespace
declarations on top. RBAC, certgen, and the controller Deployment are maintained
by upstream. EG version bumps reduce to bumping `EG_VERSION` and running
`helm dependency update`.

---

## Planned: multi-tier proxies (L1/L2)

`Gateway.spec.infrastructure.parametersRef → EnvoyProxy` (EG v1.1+) overrides
the GatewayClass-level default per Gateway. One pair can host L1 and L2 tiers:

```
GatewayClass tr-1 → EnvoyProxy/default (tr-system-1)
  Gateway/l1 in tr-dataplane-1 → EnvoyProxy/l1 (custom image, l1 filter chain)
  Gateway/l2 in tr-dataplane-1 → EnvoyProxy/l2 (l2 upstream config)
```

Each tier's proxy Deployment lands in `tr-dataplane-{i}`. EnvoyPatchPolicy
targets a specific Gateway by name -- patches are tier-scoped.

## CRD conflict strategy

| Scenario | Action |
|---|---|
| Fresh cluster | install Gateway API + EG CRDs via `gateway-crds-helm` |
| Provider-managed Gateway API (GKE, AKS) | skip Gateway API CRDs, install EG CRDs only |
| User-installed, same channel | server-side apply upgrades cleanly |
| Channel mismatch (experimental → standard) | block -- check live objects first |

Always use `gateway-crds-helm` as the single version source. EG v1.8.0 bundles
Gateway API **v1.5.1**. Install via `helm template | kubectl apply --server-side`
-- never `helm install` (Helm 1 MB release Secret limit breaks on large CRD bundles).

Provider-managed detection: `managedFields[*].manager` (NOT `bundle-version`).
Known managers: `gke-networking-controller`, `gke-gateway-api`,
`aks-gateway-api-controller`, `addon-manager`.

# Design

## Summary

`gateway-pairs` deploys N isolated Envoy Gateway controller+dataplane pairs
inside a single Kubernetes cluster. Each pair is one Helm release of `eg-pair`.
CRDs are installed once cluster-wide via `gwp crds install`.

---

## Two-namespace model per pair

```
tr-system-{i}      Helm release Secret
                   EG controller Deployment + SA + RBAC + ConfigMap

tr-dataplane-{i}   EnvoyProxy/l1, EnvoyProxy/l2-a, ...    (one per tier)
                   Gateway/l1, Gateway/l2-a, ...           (one per tier)
                   Envoy proxy Deployments + Services      (generated by EG)
                   HTTPRoutes                              (operator-managed)
```

EnvoyProxy lives in `tr-dataplane-{i}`, co-located with the Gateways.
`Gateway.spec.infrastructure.parametersRef` is a same-namespace reference;
each Gateway points at its own EnvoyProxy for per-tier customization
(custom image, dynamic modules, env vars, stable Service name).

In GatewayNamespace mode EG places the proxy Deployment in the Gateway's
namespace. Since Gateways live in `tr-dataplane-{i}`, all proxy Deployments
land there too.

The `GatewayClass` has no default `parametersRef`. Every Gateway brings
its own EnvoyProxy explicitly. This is the most flexible model and is what
[`tiered-router-eg`](https://github.com/dio/transit/tree/main/integrations/tiered-router-eg) uses.

> **Simpler single-tier deployments:** if you only need one uniform proxy,
> put a single `EnvoyProxy` in `tr-system-{i}` and wire it to
> `GatewayClass.parametersRef`. All Gateways then share the same config.
> The `eg-pair` chart ships this as a convenience default. For any real
> multi-tier deployment, ignore the chart default and apply your own
> EnvoyProxies + Gateways into `tr-dataplane-{i}`.

## Case study: [`tiered-router-eg`](https://github.com/dio/transit/tree/main/integrations/tiered-router-eg)

Reference: `https://github.com/dio/transit/tree/main/integrations/tiered-router-eg/k8s/`

This integration runs three proxy tiers under one EG controller in one pair:

```
transit-system      EG controller
transit-dataplane   EnvoyProxy/l1    (custom image + cluster-shard-router.so)
                    EnvoyProxy/l2-a  (custom image + cluster-router.so, shard a)
                    EnvoyProxy/l2-b  (custom image + cluster-router.so, shard b)
                    Gateway/l1       (infrastructure.parametersRef: EnvoyProxy/l1)
                    Gateway/l2-a     (infrastructure.parametersRef: EnvoyProxy/l2-a)
                    Gateway/l2-b     (infrastructure.parametersRef: EnvoyProxy/l2-b)
                    HTTPRoute/l1-public
                    HTTPRoute/l2-a
                    HTTPRoute/l2-b
```

L1 does tenant/cohort/BYOK sharding: maps a request to a state shard.
L2 does model+provider routing inside the shard.

Each EnvoyProxy specifies a custom Envoy image, dynamic module `.so` path, env
vars, and a stable Service name:

```yaml
apiVersion: gateway.envoyproxy.io/v1alpha1
kind: EnvoyProxy
metadata:
  name: l1
  namespace: transit-dataplane
spec:
  provider:
    type: Kubernetes
    kubernetes:
      envoyService:
        name: l1          # stable, predictable Service name
        type: ClusterIP
        labels:
          transit.dio/proxy: l1
      envoyDeployment:
        pod:
          labels:
            transit.dio/proxy: l1
        container:
          image: "custom-envoy-with-modules:tag"
          env:
          - name: ENVOY_DYNAMIC_MODULES_SEARCH_PATH
            value: /etc/envoy/dynamic-modules
  dynamicModules:
  - name: cluster-shard-router
    source:
      type: Local
      local:
        path: /etc/envoy/dynamic-modules/libcluster-shard-router.so
    doNotClose: true
```

Each Gateway references its EnvoyProxy via `infrastructure.parametersRef`:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: l1
  namespace: transit-dataplane
spec:
  gatewayClassName: tr-1
  infrastructure:
    parametersRef:
      group: gateway.envoyproxy.io
      kind: EnvoyProxy
      name: l1
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: Selector
        selector:
          matchLabels:
            transit.dio/gateway-routes: "true"
```

HTTPRoutes in `transit-dataplane` reference their Gateway by name (same namespace,
no `parentRefs.namespace` needed):

```yaml
parentRefs:
- name: l1
```

Key observations from this integration:

- `GatewayClass` has no `parametersRef`. Per-tier customization comes entirely
  from `Gateway.spec.infrastructure.parametersRef`.
- `allowedRoutes: from: Selector` uses a label on the dataplane namespace
  (`transit.dio/gateway-routes: "true"`), not `kubernetes.io/metadata.name`.
  More explicit and avoids relying on the auto-label.
- `runtimeFlags: enabled: [XDSNameSchemeV2]` is set in the EnvoyGateway config.
  Required for the transit dynamic module address scheme.
- `envoyService.name` is set explicitly (e.g. `l1`, `l2-a`) for stable,
  predictable Service names. Without this EG generates a hash-based name.

## How `eg-pair` models this

The `eg-pair` chart provides the scaffold. The operator layers tiers on top.

What `eg-pair` installs:

- `tr-system-{i}` namespace + EG controller (via `gateway-helm` subchart, planned)
- `tr-dataplane-{i}` namespace with the `tr/gateway-routes: "true"` label
- `GatewayClass tr-{i}` with unique `controllerName`
- A default single-tier `EnvoyProxy/eg` + `Gateway/eg` (ignorable for tiered deployments)

What the operator applies after install:

```bash
# Install the pair (creates controller + scaffold)
helm upgrade --install eg-pair-1 ./charts/eg-pair \
  --namespace tr-system-1 --create-namespace \
  --set pair.index=1 --skip-crds

# Layer on the tiered topology
kubectl apply -n tr-dataplane-1 -f envoyproxies-l1-l2.yaml
kubectl apply -n tr-dataplane-1 -f gateways-l1-l2.yaml
kubectl apply -n tr-dataplane-1 -f httproutes.yaml
```

The chart does not model tiers directly. The `tiers` values concept is a future
convenience wrapper; operators can always apply raw manifests.

## Layering: cluster, pair, tenant

Everything in `gateway-pairs` falls into one of three layers. Understanding
which layer a resource belongs to determines who installs it, who owns it,
and what breaks if you get it wrong.

### Layer 1: cluster-wide (installed once)

**Gateway API CRDs** and **EG CRDs** are cluster-scoped and version-sensitive.
They must be installed before any pair. One installation serves all pairs.

```
GatewayClass              (cluster-scoped, but per-pair -- see Layer 2)
CustomResourceDefinitions (cluster-scoped, shared -- this is the one-time install)
```

`gwp crds install` handles this. It detects whether the cluster already has
Gateway API CRDs (user-installed or provider-managed), installs only what is
missing, and validates that the installed version is compatible with the EG
version being deployed.

Provider-managed Gateway API (GKE, AKS autopilot) means `gwp crds install`
skips Gateway API CRDs but still installs EG CRDs. See the CRD conflict
strategy section below.

### Layer 2: per pair (one `eg-pair` Helm release)

Each `helm upgrade --install eg-pair-{i}` installs exactly one pair. Within
this layer there are two sub-scopes:

**Cluster-scoped, but unique per pair:**

| Resource | Why unique |
|---|---|
| `GatewayClass tr-{i}` | GatewayClass names are cluster-scoped; two pairs cannot share one without their controllers fighting |
| `ClusterRole eg-pair-tr-{i}-tokenreviews` | Bound to this pair's controller SA |
| `ClusterRole eg-pair-tr-{i}-gateway-controller` | Same |
| `ClusterRoleBinding` (both) | Binds to this pair's `envoy-gateway` SA in `tr-system-{i}` |

N pairs in one cluster means N GatewayClasses and N sets of ClusterRoles.
The naming scheme (`eg-pair-tr-{i}-*`) prevents collisions.

**Namespace-scoped (pair's two namespaces):**

Everything else in the pair is namespace-scoped and naturally isolated because
`tr-system-{i}` and `tr-dataplane-{i}` are unique to this pair.

### Layer 3: per tenant/integration (operator-applied)

This is where `tiered-router-eg` lives. After `eg-pair-{i}` is installed,
the operator applies the integration's resources into `tr-dataplane-{i}`:

```bash
kubectl apply -n tr-dataplane-1 -f envoyproxies-l1-l2.yaml
kubectl apply -n tr-dataplane-1 -f gateways-l1-l2.yaml
kubectl apply -n tr-dataplane-1 -f httproutes.yaml
```

The only coupling between Layer 3 and Layer 2 is the `gatewayClassName` field
in each Gateway manifest. It must reference the pair's GatewayClass:

```yaml
spec:
  gatewayClassName: tr-1    # must match the GatewayClass created by eg-pair-1
```

Everything else in Layer 3 is self-contained. Multiple `tiered-router-eg`
tenants in the same cluster each get their own pair (Layer 2 install) and
their own `tr-dataplane-{i}` namespace into which they apply their topology.

### The full picture for N tenants

```
Cluster
  Layer 1 (once):
    Gateway API CRDs (v1.5.1)
    EG CRDs (v1.8.0)

  Layer 2 (per tenant, via eg-pair):
    GatewayClass tr-1          (cluster-scoped)
    ClusterRole/Binding tr-1   (cluster-scoped)
    tr-system-1/               (controller)
    tr-dataplane-1/            (scaffold + RBAC, label: tr/gateway-routes=true)

    GatewayClass tr-2
    ClusterRole/Binding tr-2
    tr-system-2/
    tr-dataplane-2/

  Layer 3 (per tenant, operator-applied):
    tr-dataplane-1/
      EnvoyProxy/l1, EnvoyProxy/l2-a, EnvoyProxy/l2-b
      Gateway/l1 (gatewayClassName: tr-1)
      Gateway/l2-a, Gateway/l2-b
      HTTPRoutes

    tr-dataplane-2/
      EnvoyProxy/l1, EnvoyProxy/l2-a, ...
      Gateway/l1 (gatewayClassName: tr-2)
      ...
```

Tenant 1's controller (`tr-system-1`) only watches `tr-system-1` and
`tr-dataplane-1`. It will never see tenant 2's resources, even though their
Gateways reference the same EG version and run in the same cluster.

## Resources by scope
|---|---|---|
| Helm release Secret | tr-system-{i} | Helm |
| Namespace tr-system-{i} | cluster | `--create-namespace` + chart resource |
| Namespace tr-dataplane-{i} | cluster | chart template |
| GatewayClass tr-{i} | cluster-scoped | chart template |
| ClusterRole/Binding tokenreviews | cluster-scoped | chart template |
| ClusterRole/Binding gateway-controller | cluster-scoped | chart template |
| ServiceAccount envoy-gateway | tr-system-{i} | chart template |
| Deployment envoy-gateway (controller) | tr-system-{i} | chart template |
| Service envoy-gateway (xDS + metrics) | tr-system-{i} | chart template |
| ConfigMap envoy-gateway-config | tr-system-{i} | chart template |
| Role/RoleBinding infra-manager | tr-system-{i} | chart template |
| Role/RoleBinding infra-manager | tr-dataplane-{i} | chart template |
| EnvoyProxy (l1, l2-a, l2-b, ...) | tr-dataplane-{i} | operator-managed |
| Gateway (l1, l2-a, l2-b, ...) | tr-dataplane-{i} | operator-managed |
| Envoy proxy Deployments + Services | tr-dataplane-{i} | EG controller (generated) |
| HTTPRoutes | tr-dataplane-{i} | operator-managed |

> The chart also creates `EnvoyProxy/eg` in `tr-system-{i}` and `Gateway/eg`
> in `tr-dataplane-{i}` as a single-tier convenience default. Operators running
> a tiered topology can disable these via `gateway.create: false` and
> `envoyProxy.create: false` (planned values).

## Namespace naming

All names derive from `pair.namePrefix`, `pair.index`, and `pair.nameSuffix`.

Rules:

- `namePrefix`: auto-appends `-` if non-empty and not already ending with one.
- `nameSuffix`: appended as-is. The caller controls the separator.
- `nameSuffix` empty + `index > 0`: suffix defaults to `-{index}`.
- `nameSuffix` empty + `index == 0`: no suffix (single/unnamed pair).
- GatewayClass name omits the role fragment, just prefix+suffix.

| namePrefix | index | nameSuffix | systemNS | dataplaneNS | GatewayClass |
|---|---|---|---|---|---|
| `tr` | `1` | `""` | `tr-system-1` | `tr-dataplane-1` | `tr-1` |
| `tr` | `0` | `""` | `tr-system` | `tr-dataplane` | `tr` |
| `tr` | `0` | `-prod` | `tr-system-prod` | `tr-dataplane-prod` | `tr-prod` |
| `tr` | `0` | `-eu-west` | `tr-system-eu-west` | `tr-dataplane-eu-west` | `tr-eu-west` |
| `""` | `1` | `""` | `system-1` | `dataplane-1` | `1` |

## allowedRoutes wiring

The chart labels `tr-dataplane-{i}` with `tr/gateway-routes: "true"`. Gateway
listeners use `from: Selector` targeting that label:

```yaml
allowedRoutes:
  namespaces:
    from: Selector
    selector:
      matchLabels:
        tr/gateway-routes: "true"
```

HTTPRoutes in the same namespace reference the Gateway by name only:

```yaml
parentRefs:
- name: l1
```

---

## Controller isolation: unique controllerName per pair

Each pair's EG controller must have a unique `controllerName`. The chart derives
it from the GatewayClass name:

```yaml
# ConfigMap envoy-gateway-config:
gateway:
  controllerName: gateway.envoyproxy.io/tr-1

# GatewayClass:
spec:
  controllerName: gateway.envoyproxy.io/tr-1
```

Without this, controllers fight over each other's GatewayClasses and log
`failed to find envoyproxy: unknown namespace for the cache` constantly.

## Watch list

Both namespaces must appear in the watch list:

```yaml
watch:
  type: Namespaces
  namespaces:
  - tr-system-1     # controller reads its TLS secret and manages infra here
  - tr-dataplane-1  # controller manages Gateways, EnvoyProxies, proxy resources
```

---

## Authentication

Default EG mode uses mTLS. Gateway Namespace mode shifts to server-side TLS
with JWT token validation. Proxy pods in `tr-dataplane-{i}` authenticate using
projected ServiceAccount JWT tokens. The controller validates them via
`TokenReview`, which requires `tokenreviews/create` at cluster scope. Without
it, proxies stay unready and the controller logs
`tokenreviews.authentication.k8s.io is forbidden`.

## Uninstall sequence

Delete Gateways before uninstalling. The proxy pod carries a finalizer that
only the running controller can clear. Removing the controller first leaves the
finalizer uncleared and the namespace hangs in Terminating indefinitely.

```bash
# Delete Gateways (EG deprovisions proxies and clears finalizers)
kubectl delete gateway --all -n tr-dataplane-1 --wait=true --timeout=60s

# Wait for proxy Deployments to be gone
kubectl wait --for=delete deployment \
  -l app.kubernetes.io/managed-by=envoy-gateway \
  -n tr-dataplane-1 --timeout=60s

# Helm uninstall removes everything tracked in the release
helm uninstall eg-pair-1 -n tr-system-1
```

Both `tr-system-1` and `tr-dataplane-1` terminate cleanly after this sequence.

## Planned: subchart migration

Add `gateway-helm` as a Helm subchart dependency. All required values are
settable as plain `--set` flags (confirmed against v1.8.0):

```
config.envoyGateway.gateway.controllerName           ✓
config.envoyGateway.provider.kubernetes.deploy.type  ✓
config.envoyGateway.provider.kubernetes.watch.*      ✓
topologyInjector.enabled                             ✓
```

`eg-pair` becomes a thin wrapper: passes these values to `gateway-helm` and
adds namespace declarations, GatewayClass, and the default single-tier scaffold.
RBAC, certgen, and the controller Deployment are maintained by upstream.

## Merged Gateways incompatibility

Gateway Namespace Mode is not supported with Merged Gateways deployments.

## CRD conflict strategy

| Scenario | Action |
|---|---|
| Fresh cluster | `gwp crds install` |
| Provider-managed Gateway API (GKE, AKS) | `gwp crds install --skip-gateway-api-crds` |
| User-installed, same channel | `gwp crds install` (server-side apply is idempotent) |
| Channel mismatch (experimental to standard) | `gwp crds detect` first; block if live objects exist |

`gwp crds install` runs `helm template | kubectl apply --server-side` internally.
Direct `helm install` is never used because Helm's 1 MB release Secret limit
breaks on large CRD bundles. `gwp crds detect` inspects
`managedFields[*].manager` to identify provider-managed installs and reports
compatibility against the bundled Gateway API version.
